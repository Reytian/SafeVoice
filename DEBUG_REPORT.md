# SafeVoice — Debug Report

_Generated by the `debug-safevoice` multi-agent workflow (run `wf_3a668637-1f0`)._

**46 candidates checked → 30 confirmed → 27 after dedupe.**  
Severity: 0 critical, 6 high, 13 medium, 8 low.

## Overall assessment

SafeVoice's core voice pipeline is functional, but its error-handling and concurrency discipline are weak, and several realistic, non-exotic conditions can either permanently wedge the app or silently lose the user's transcript. The most serious cluster is a single design flaw repeated across lanes: _start_listening() commits app state (STATE_LISTENING) and HotkeyManager state (_is_active) BEFORE AudioCapture.start() succeeds, and AudioCapture.start()/stop() raise on the unhappy path. A transient mic-busy / device-unplugged event (extremely common on macOS with AirPods/USB mics) therefore bricks the app until restart AND leaks a PortAudio stream. A second cluster is a thread-safety violation: ASR is invoked concurrently by the speculative and final transcribe threads against one unlocked MLX session (Metal crash / corruption), and the menubar status title/menu-item is mutated from background threads on every transcription, violating the same off-main-thread-AppKit rule the project documents as a real bug and already solves correctly in overlay.py. A third cluster is data loss: user-defined snippets with a backslash crash every dictation (total transcript loss), a corrupt vocabulary.json crashes app startup, the English stutter regex corrupts repeated digits (e.g. spoken PINs) and CJK reduplication, injection failures are recorded as successes while the text is lost, and 512-token truncation can silently drop the tail of long dictations. Pervasive silent error-swallowing (settings load/save, history writes surfacing as fake transcription errors, custom-mode LLM failures indistinguishable from no-ops) makes all of this hard to diagnose, directly defeating the project's stated overnight-debugging goal. None of these are style nits; the top ~8 are crash/wedge/data-loss paths that a normal user will hit. Recommended priorities: (1) make audio start/stop failure-safe and revert state to IDLE; (2) serialize ASR access with a lock and/or join the speculative thread; (3) route all status-bar UI through the main thread; (4) harden the two JSON loaders and the snippet replacement; (5) treat injection-failure and LLM-truncation as failures, not successes.

## Ranked findings

### #1 · [HIGH] Failed audio start permanently wedges the app (state stuck) and audio.stop() then raises
- **File:** `src/app.py (+ src/audio_capture.py)`  ·  **Lines:** app.py 580-597, 608-629; audio_capture.py 120-122  ·  **Category:** crash
- **What:** _start_listening() sets STATE_LISTENING and shows the overlay (app.py:582,588) BEFORE calling self._audio.start() (app.py:591). AudioCapture.start() raises sd.PortAudioError/RuntimeError on a realistic transient (mic held by another app, AirPods/USB mic unplugged, no/changed default input device, missing mic TCC permission, post-sleep Core Audio failure). _on_hotkey_activate only logs it, leaving _state==STATE_LISTENING and HotkeyManager._is_active==True. On key release _on_hotkey_deactivate passes its STATE_LISTENING guard, _stop_listening_and_transcribe sets STATE_TRANSCRIBING and calls self._audio.stop() at app.py:629, which raises RuntimeError('Recording is not in progress.') because start() never succeeded. The transcribe thread never starts, so its finally{_set_state(STATE_IDLE)} never runs: state is pinned at TRANSCRIBING, the overlay never hides, all future hotkey presses are ignored (activate guard requires STATE_IDLE), and the feature is dead until restart. The intended transcript is lost with no user-visible error.
- **Fix:** In _start_listening, wrap self._audio.start() in try/except; on failure call _set_state(STATE_IDLE), hide the overlay, stop the speculative timer, and surface an error, returning without leaving STATE_LISTENING. Prefer setting STATE_LISTENING / showing the overlay only AFTER start() succeeds. Additionally guard _stop_listening_and_transcribe with `if not self._audio.is_recording: self._set_state(STATE_IDLE); return` (or wrap self._audio.stop() in try/except RuntimeError) so stop() with no active stream can never raise.

### #2 · [HIGH] Speculative ASR thread and final transcribe thread call the same MLX session concurrently
- **File:** `src/app.py (+ src/asr_engine.py)`  ·  **Lines:** app.py 641, 743-767; asr_engine.py single shared _session, no lock  ·  **Category:** race
- **What:** In any custom mode (prompt_template set) with the LLM available, _start_listening starts a speculative worker (_speculate) that periodically calls self._asr.transcribe() (app.py:755). _stop_speculative_timer() (app.py:765-767) only sets a threading.Event and does NOT join the worker, which is very likely still blocked inside MLX inference (hundreds of ms to seconds). Immediately afterward the final transcribe thread calls self._asr.transcribe() (app.py:641) on the SAME stateful mlx Session, which has no lock. MLX/Metal sessions are not reentrant; two concurrent transcribe() calls race on shared graph/KV-cache state and can crash the process, corrupt the returned text, or wedge the session. Triggered by holding the hotkey past one speculative tick in a custom mode, then releasing while inference is in flight.
- **Fix:** Serialize all ASR access: add a threading.Lock to ASREngine and acquire it inside transcribe()/feed_chunk()/finish_streaming(). Additionally (or alternatively) store the speculative Thread in self._speculative_thread and join it with a timeout in _stop_speculative_timer() before starting the final transcribe thread.

### #3 · [HIGH] apply_snippets crashes on backslash/group-ref in a user snippet replacement, losing the whole transcript
- **File:** `src/vocabulary.py`  ·  **Lines:** 59-63 (call site app.py:645)  ·  **Category:** crash
- **What:** apply_snippets escapes the trigger with re.escape() but passes the raw user replacement to pattern.sub() (vocabulary.py:62), so re.sub interprets it as a replacement template. A normal snippet such as a Windows path 'C:\Users\me' raises re.error('bad escape \U'); a replacement containing '\1' raises 'invalid group reference 1'; 'a\b' silently drops content. apply_snippets runs inside transcribe()'s try block (app.py:645), so once any such snippet exists in ~/.config/safevoice/vocabulary.json, EVERY subsequent dictation throws, is swallowed by the handler at app.py:725 (shows 'Error: ...' for 1s), and the transcript is never pasted or saved to history. Total recurring data loss until the snippet is deleted.
- **Fix:** Do not let re.sub interpret the replacement. Use a function replacement: pattern.sub(lambda _m: replacement, text). (Equivalently escape it: replacement.replace('\\','\\\\').) The lambda form treats the replacement as a literal and fixes both the crash and the silent-corruption cases.

### #4 · [HIGH] VocabularyManager._load crashes app startup on corrupt / non-UTF-8 / non-dict vocabulary.json
- **File:** `src/vocabulary.py`  ·  **Lines:** 20-25 (constructed unconditionally at app.py:139)  ·  **Category:** crash
- **What:** _load() opens vocabulary.json and calls json.load() with no try/except, no encoding argument, and assumes a dict (data.get(...)). A truncated write (the non-atomic _save at vocabulary.py:27-29 interrupted by quit/crash/power loss), a hand-edited file, a non-UTF-8 file, or a top-level array/string raises JSONDecodeError/UnicodeDecodeError/AttributeError directly out of __init__. Because VocabularyManager() is built unconditionally at app.py:139, the menubar app fails to launch (sys.exit(1), only a /tmp/safevoice.log entry); recovery requires manually deleting the file, which a non-technical user cannot discover.
- **Fix:** Wrap the read/parse in try/except (json.JSONDecodeError, OSError, UnicodeDecodeError, ValueError), pass encoding='utf-8' to open(), validate that data is a dict (hotwords a list, snippets a dict), and on any failure log a warning and fall back to empty defaults. Optionally make _save atomic (write tmp + os.replace) to prevent truncation.

### #5 · [HIGH] Audio InputStream leaked when stream open/start fails (and on every subsequent retry)
- **File:** `src/audio_capture.py`  ·  **Lines:** 99-107  ·  **Category:** leak
- **What:** start() assigns the new stream to self._stream BEFORE calling self._stream.start() (audio_capture.py:99-107). If the InputStream opens but start() raises (device busy/unavailable, unsupported 16kHz/mono, permission), the exception propagates with self._stream set to the opened-but-failed stream and self._recording==False. _on_quit only calls stop() when is_recording is True (app.py:792), so the opened PortAudio stream/device handle is never closed; there is no __del__/atexit covering it. A later start() overwrites self._stream with a fresh stream, orphaning the previous one entirely. This is the resource-leak facet of the same failed-start scenario as rank 1 (mic hot-swap on macOS).
- **Fix:** Build the stream in a local variable, start it, and only then assign self._stream and set self._recording=True. On any exception, call stream.close() and reset self._stream=None/self._recording=False before re-raising: `stream = sd.InputStream(...)`; `try: stream.start()` `except Exception: stream.close(); raise`; `self._stream = stream; self._recording = True`.

### #6 · [MEDIUM] Transcript silently lost when injection fails (no permission / paste fails) but recorded as success
- **File:** `src/app.py`  ·  **Lines:** 707-720, 769-783  ·  **Category:** data-loss
- **What:** In transcribe(), the overlay is hidden (app.py:709) and then _inject_text(text) is called (711). _inject_text returns None in two failure cases: no Accessibility permission (early return at 777, no paste) and self._injector.inject() returning False (only a print at 783; inject() returns False e.g. when CGEventCreateKeyboardEvent returns None under memory pressure, or clipboard write fails). The caller ignores the outcome and unconditionally runs record_transcription (712) and history.add (714) as if the paste succeeded. The text is pasted nowhere and the clipboard is restored ~0.5s later, so the transcript is fully lost, yet stats and history count it as successful. The only 'Grant Accessibility permission' message (775) is written to an overlay that was already hidden at 709, so it is invisible. Permission can also be revoked mid-flight (OS update / TCC reset) between grant and paste.
- **Fix:** Make _inject_text return a bool (False on missing permission and on inject()==False). At the call site, only record stats/history when injection succeeded; on failure re-show the overlay with a clear error ('Paste failed - text is on the clipboard' / 'Grant Accessibility permission') and keep the transcript on the clipboard. Move the permission check before overlay.hide(), or do not hide before showing the warning.

### #7 · [MEDIUM] Menubar status title/menu-item mutated from background threads (off-main-thread AppKit)
- **File:** `src/app.py`  ·  **Lines:** 530-548, 331-355, 611-612, 678, 733-734, 771  ·  **Category:** race
- **What:** _set_state writes self.title (app.py:541) and _update_status writes self._status_item.title (546). For rumps these are direct AppKit calls (NSStatusItem.setStatusBarTitle / NSMenuItem.setTitle_). Both are invoked from background threads on every normal path: the model-load daemon thread (336/339/353), the transcribe worker (611/733), and _inject_text (771). AppKit/Cocoa object mutation off the main thread is undefined behavior and is the exact hazard overlay.py already avoids via performSelectorOnMainThread. It is a race (not deterministic) but the off-main-thread access happens on 100% of transcriptions and model loads; under concurrent main-thread drawing (status menu open / redraw) it can corrupt the title or crash at the ObjC level. _update_status swallows exceptions (545-548) but _set_state has no guard.
- **Fix:** Route _set_state's title write and _update_status's menu-item write through the main thread, reusing overlay.py's NSObject-trampoline + performSelectorOnMainThread_withObject_waitUntilDone_(...,False) pattern (or a libdispatch main-queue hop). Never touch self.title / self._status_item.title directly from background threads.

### #9 · [MEDIUM] HistoryStore.add / post-paste bookkeeping failures surface as a misleading transcription 'Error'
- **File:** `src/app.py (+ src/history.py)`  ·  **Lines:** app.py 711-731; history.py 44-57  ·  **Category:** silent-failure
- **What:** After _inject_text successfully pastes (app.py:711), record_transcription (712) and history.add (714) run inside the same broad try/except (725-731) that shows a red 'Error: <msg>' overlay and logs 'Transcription error'. ### #10 · [MEDIUM] English stutter regex corrupts repeated digits (spoken PINs/numbers) and CJK reduplication
- **File:** `src/text_postprocess.py`  ·  **Lines:** 60, 105  ·  **Category:** data-loss
- **What:** _EN_STUTTER_RE = \b(\w+)(?:\s+\1\b)+ collapses any whitespace-separated repeated token, case-insensitively. Python \w matches digits and CJK. Empirically: 'my pin is 1 1 2 4' -> 'my pin is 1 2 4' (wrong PIN), '好 好 学习' -> '好 学习' (destroys 好好学习). It also collapses legitimate English reduplication ('very very good' -> 'very good', 'no no no' -> 'no', 'bye bye' -> 'bye'). strip_filler_words runs on the no-LLM path (app.py:700) AND as the pre-clean inside llm_cleanup (llm_cleanup.py:312), so the dropped digit/token is also what gets sent to the LLM, making the loss unrecoverable even with Ollama running.
- **Fix:** Restrict the spaced-duplicate collapse to ASCII alphabetic words and short lengths (e.g. require [A-Za-z]+ with length <= ~3) and exclude pure-digit groups; do not apply the spaced-duplicate rule across CJK (CJK reduplication is already handled conservatively by _CN_STUTTER_RE). At minimum add digit / CJK negative-lookahead guards so numbers and Chinese are never collapsed.

### #12 · [MEDIUM] 512-token LLM truncation can paste a mid-sentence-cut cleanup undetected on long dictation
- **File:** `src/llm_backend.py (+ src/llm_cleanup.py)`  ·  **Lines:** llm_backend.py 227,313,359,393; llm_cleanup.py 361  ·  **Category:** data-loss
- **What:** All backends cap generation at 512 tokens (Ollama num_predict, MLX max_tokens, cloud max_tokens). The runaway-length guard (llm_cleanup.py:361) is character-based (reject > 2*len(input)+60). For a long dictation (no duration cap; recorder runs until release) the truncated-at-512-tokens output is SHORTER than the input, so G1 never fires, and its length ratio stays above 0.6 so the over-deletion guard never fires either. The mid-sentence-truncated rewrite passes both guards, is pasted, and saved to history; the user loses the tail of their dictation with no warning (especially for CJK, where 512 tokens is well below a long passage's character count).
- **Fix:** Raise num_predict/max_tokens to comfortably exceed expected dictation (e.g. 1024-2048) and/or detect truncation: Ollama returns done_reason; cloud returns finishReason. If the response was truncated (done_reason=='length' / finishReason in {MAX_TOKENS,length}), log a warning and fall back to pre_cleaned so the full rule-stripped transcript is preserved instead of a truncated rewrite.

### #13 · [MEDIUM] Settings load/save swallow all errors with no logging (disk-full / permission / corrupt file)
- **File:** `src/settings_manager.py`  ·  **Lines:** 272-273, 288-289  ·  **Category:** silent-failure
- **What:** _load() does `except (json.JSONDecodeError, OSError): pass` and _save_locked() does `except OSError: pass`, with no logging anywhere in the module. _save_locked is the only persistence path for both settings and usage stats (set, set_many, reset_to_defaults, record_transcription, migration). If the config dir is unwritable or the disk is full, every change is silently discarded (appears applied in memory, reverts on next launch) with zero diagnostics; a corrupt settings.json silently resets all settings to defaults. This violates the 'never silently swallow errors' rule and defeats the project's overnight-diagnosis goal.
- **Fix:** Add a module logger and log with context in both handlers (logger.exception('Failed to save settings to %s', self._path) / 'Failed to load settings'). Keep the app running but make the failure diagnosable; optionally also catch the json/TypeError serialization case in _save_locked.

### #14 · [LOW] Custom-mode result discarded when equal to input, and silent LLM failure is indistinguishable from a real no-op

### #15 · [MEDIUM] Custom-prompt cleanup path bypasses all safety guards (runaway-length / over-deletion / think-leak)
- **File:** `src/llm_cleanup.py`  ·  **Lines:** 317-330  ·  **Category:** silent-failure
- **What:** When custom_prompt is set, cleanup() calls self._backend.chat(...) and returns the result directly if truthy (llm_cleanup.py:317-330), skipping all three guards the default path applies (runaway-length at 361, over-deletion at 378, script/mixed-script at 338/348). - **Fix:** Factor the runaway-length (and ideally over-deletion) guards into a helper and apply them on the custom-prompt branch too: after result = self._backend.chat(...), if the runaway-length check trips (relative to the input/prompt length) log and return pre_cleaned instead of result.

### #17 · [MEDIUM] Toggle-mode desync: a mistimed hotkey press during transcription leaves HotkeyManager._is_active stuck

### #21 · [MEDIUM] Fixed 0.5s clipboard-restore delay can race an async Cmd+V and paste the wrong clipboard
